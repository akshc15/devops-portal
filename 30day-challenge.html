The Big Picture — Explained Like You're 5
Before any commands, understand WHAT you're building and WHY.
🧒 Imagine this story:

You bake a cake (your app). Right now the cake only works in YOUR kitchen (your laptop). Docker puts the cake in a lunchbox — now it works in ANY kitchen, anywhere.

But what if millions of people want cake? You need a factory. Kubernetes is that factory — it runs hundreds of lunchboxes, replaces broken ones automatically, and scales up when orders spike.

And instead of manually baking every time you change the recipe, you set up a robot assembly line. That's CI/CD (GitHub Actions) — every time you push code, the robot tests it, bakes it, puts it in a lunchbox, and ships it to the factory. Automatically.

That's the entire DevOps cycle. In 30 days, you'll build all of this.

Install Everything — Step by Step
From scratch. Where to click. What to type. Nothing assumed.
🐳 Docker Desktop Step 1

Docker Desktop runs Docker on your Mac/Windows. It gives you a nice UI and the docker command in terminal.

Where to go: Open Chrome → go to docker.com/products/docker-desktop → click the big green Download button → pick Mac (Apple chip) or Mac (Intel) → open the .dmg file → drag Docker to Applications → open Docker from Applications → wait for the whale icon to stop moving in your menu bar ✓
docker --version # Should print: Docker version 25.x.x
☸️ Minikube (local K8s) Step 2

Minikube creates a tiny Kubernetes cluster on your own laptop. Free. No cloud needed to learn.

Mac (Homebrew): Open Terminal → paste these commands one by one:
/bin/bash -c "$(curl -fsSL https://raw.githubusercontent.com/Homebrew/install/HEAD/install.sh)" brew install minikube brew install kubectl minikube start
Windows: Go to minikube.sigs.k8s.io → Installation → Windows → download the .exe → run it → open new terminal → type minikube start
🐙 Git + GitHub Step 3

Git saves versions of your code. GitHub stores it online and runs your CI/CD pipeline.

Install Git: Mac → open Terminal → type git --version (Mac auto-installs it). Windows → go to git-scm.com → Download → run installer → all defaults → Finish.
GitHub account: Go to github.com → Sign up (free) → verify email → done.
git config --global user.name "Your Name" git config --global user.email "[email protected]"
🐍 Python + FastAPI Step 4

This is the app we'll build, containerize, and deploy across all 30 days — TaskAPI.

Install Python: Go to python.org/downloads → click yellow Download button → run installer → ✅ tick "Add Python to PATH" (important!) → Install Now.
python --version # should say 3.11+ pip install fastapi uvicorn # Now you can run the app!
⛵ Helm Step 5 — Week 4

Install in Week 4. Helm is the package manager for Kubernetes — like npm but for K8s apps.

Mac: Open Terminal → brew install helm
Windows: Open PowerShell as admin → choco install kubernetes-helm
helm version # version.BuildInfo{Version:"v3.x.x"...}
🔑 GitHub Secrets (CI/CD) Step 6 — Week 3

Your pipeline needs passwords (Docker Hub, K8s config) stored safely — not in your code.

Where to click: Go to your GitHub repo → click Settings tab → left sidebar click Secrets and variables → Actions → click New repository secret → Name: DOCKERHUB_USERNAME → paste your Docker Hub username → Add secret. Repeat for DOCKERHUB_TOKEN (get token from hub.docker.com → Account Settings → Security → New Access Token).

Top Interview Questions — With Answers
Click any question to reveal the answer. Real questions from real DevOps interviews.
Q1What is the difference between a Docker image and a container?
Image = the blueprint (read-only). Like a class in programming, or a cake recipe. Container = a running instance of that image. Like an object, or a baked cake. You can run 10 containers from 1 image simultaneously. Images are built once, containers are created and destroyed.
Say: "Image is immutable and stored in a registry. Container is ephemeral and runs in memory."
Q2What is Kubernetes and why do we need it?
Docker runs one container on one machine. Kubernetes orchestrates thousands of containers across hundreds of machines. It handles: scheduling (which container runs where), self-healing (restarts crashed pods), scaling (adds pods when traffic spikes), rolling updates (zero-downtime deploys), and service discovery (pods find each other by name).
Say: "Docker solves packaging. Kubernetes solves running at scale."
Q3What is a Pod in Kubernetes?
The smallest deployable unit in Kubernetes. A Pod wraps one or more containers that share the same network namespace and storage. Think of it as a "logical host" — containers inside a Pod talk to each other via localhost. In practice, most Pods have one container. K8s doesn't manage containers directly — it manages Pods.
Say: "Pod is to K8s what container is to Docker — the basic unit."
Q4What is the difference between a Deployment and a Service?
Deployment manages Pods — how many replicas, what image to run, how to update. It ensures desired state matches actual state. Service exposes those Pods to the network — gives them a stable IP and DNS name. Pods come and go (they're ephemeral), but a Service always points to the right Pods. Without a Service, you can't reliably reach your Deployment.
Say: "Deployment = what runs. Service = how to reach it."
Q5What is CI/CD? Explain the full pipeline.
CI (Continuous Integration): Every code push automatically triggers: code checkout → install deps → run tests → build Docker image. If tests fail, the pipeline stops. No broken code moves forward. CD (Continuous Delivery/Deployment): On success: push image to registry → update Kubernetes manifest → kubectl apply → rolling update deploys new version. Full pipeline: git push → tested → built → deployed in ~90 seconds with zero manual steps.
Say: "CI catches bugs. CD ships fast. Together: safe, rapid delivery."
Q6What is a liveness probe vs readiness probe?
Liveness probe: "Is this container alive?" K8s sends an HTTP GET to /health every N seconds. If it fails, K8s kills and restarts the container. Fixes deadlocks and stuck processes. Readiness probe: "Is this container ready to receive traffic?" If it fails, K8s removes the Pod from the Service's endpoints — it won't receive requests until it's ready again. This prevents sending traffic to a starting-up or overloaded Pod.
Say: "Liveness = restart if sick. Readiness = stop traffic if not ready."
Q7How do you roll back a bad deployment in Kubernetes?
Kubernetes stores deployment history. Three ways: 1. kubectl rollout undo deployment/taskapi — reverts to previous version instantly. 2. kubectl rollout undo deployment/taskapi --to-revision=2 — go to specific revision. 3. Update the image tag in your manifest to a known-good version and kubectl apply. Always use image tags (not :latest) so you know exactly which version is running.
Say: "undo is instant. For production, I'd update the image tag and apply, so the rollback is tracked in git."
Q8What is Helm and why use it?
Helm is the package manager for Kubernetes. A Helm chart bundles all your K8s YAML files into one reusable package with variables. Without Helm: 6 YAML files, manually update image tags, duplicate configs for staging/prod. With Helm: one values.yaml file to customise, helm install taskapi ./chart deploys everything, helm upgrade taskapi ./chart --set image.tag=v2 updates. Also handles rollback, versioning, and templating across environments.
Say: "Helm solves configuration management for K8s — it's like npm for your infrastructure."

6 Mistakes That Kill Junior DevOps Engineers
Learn these now. They come up in every interview and cost hours in production.
01Using :latest image tag in production

":latest" means "whatever was last pushed" — nobody knows which version is actually running. When a bug appears, you can't roll back to "the previous latest".

Use commit SHA tags: image: yourusername/taskapi:a3f2c91
02Hardcoding secrets in YAML files

If you put passwords or API keys in your K8s manifests or GitHub workflow files, they end up in git history — visible to everyone forever.

Use GitHub Secrets + K8s Secrets objects. Never commit credentials.
03No resource limits on pods

One runaway pod can consume all CPU/memory on the node, crashing every other pod on that machine. This is called the "noisy neighbour" problem.

Always set requests AND limits for cpu and memory in every deployment.
04Running containers as root

If an attacker exploits your app inside a container running as root, they effectively have root on the container and potentially the host node.

Add USER appuser in Dockerfile. Add securityContext: runAsNonRoot: true in K8s.
05EXPOSE in Dockerfile doesn't publish ports

EXPOSE is documentation only — it tells humans what port the app uses but does NOT actually expose it. Beginners waste hours wondering why they can't connect.

Use -p 8000:8000 in docker run OR ports: in docker-compose to actually publish.
06No health probes = silent failures

Without liveness probes, K8s doesn't know your app is stuck in a deadlock. It keeps sending traffic to a frozen container thinking it's healthy.

Add livenessProbe + readinessProbe pointing to /health in every deployment.yaml.
