The Big Picture — Explained Like You're 5
If you understand this story, everything else is just filling in details.
👶 The Story:

You write an app on your laptop. Works great! But your friend's laptop? Crashes. Server? Breaks. Docker fixes this — it packs your app + everything it needs into a sealed lunchbox that runs identically everywhere.

Now your app is popular — 10,000 users hit it at once. One lunchbox isn't enough. You need 50 running in parallel, someone to restart broken ones, someone to route traffic. That's Kubernetes — the factory manager that runs thousands of lunchboxes automatically.

You need a server to run Kubernetes on. Instead of clicking around AWS for 2 hours, you write a code file describing what you want. Terraform reads it and creates the server in 3 minutes. Repeatable. Version controlled.

Every time you change code and push to GitHub, a robot (GitHub Actions/Jenkins) automatically tests it, builds it, containerises it, and deploys it. No manual steps. That's CI/CD.

Is everything healthy? Is there a memory leak? Did 500 errors spike? Prometheus + Grafana watch everything and show you live dashboards.

That's DevOps. You just understood it completely. Everything else is details.

SCRIPTING 📝 Shell Script — Your First Automation Script

Shell scripts are just text files with commands. Save them with .sh extension. Make executable. Run them.

Create: nano deploy.sh
#!/bin/bash
# deploy.sh — builds, tests, and runs TaskAPI
# #!/bin/bash tells Linux: use bash to run this file

set -e  # stop script immediately if any command fails

APP_NAME="taskapi"
IMAGE_TAG="latest"

echo "🚀 Starting deployment of $APP_NAME..."

# Functions — reusable blocks of commands
check_docker() {
  if ! command -v docker &> /dev/null; then
    echo "❌ Docker not installed. Run setup script first."
    exit 1
  fi
  echo "✓ Docker found: $(docker --version)"
}

run_tests() {
  echo "🧪 Running tests..."
  python3 -m pytest tests/ -v
  echo "✓ All tests passed"
}

build_image() {
  echo "🐳 Building Docker image..."
  docker build -t $APP_NAME:$IMAGE_TAG .
  echo "✓ Image built: $APP_NAME:$IMAGE_TAG"
}

deploy() {
  echo "🔄 Stopping old container if running..."
  docker stop $APP_NAME 2>/dev/null || true
  docker rm $APP_NAME 2>/dev/null || true

  echo "▶️  Starting new container..."
  docker run -d \
    --name $APP_NAME \
    -p 8000:8000 \
    --restart unless-stopped \
    $APP_NAME:$IMAGE_TAG

  echo "✓ $APP_NAME is running at http://localhost:8000"
}

# Run all functions in order
check_docker
run_tests
build_image
deploy

echo ""
echo "✅ Deployment complete!"
echo "   Test it: curl localhost:8000/health"
run the script
$ chmod +x deploy.sh
$ ./deploy.sh

DOCKERFILE 🐳 TaskAPI Dockerfile — Every Line Explained 
# FROM — which base image to start from
# "slim" = smaller image, fewer attack surfaces
FROM python:3.11-slim

# WORKDIR — set working directory inside container
# All following commands run from /app
WORKDIR /app

# COPY requirements BEFORE the app code
# Docker caches layers — if requirements didn't change,
# it skips the pip install step on rebuild (faster!)
COPY requirements.txt .

# RUN — execute commands during BUILD time
# --no-cache-dir = don't store pip cache in image (smaller)
RUN pip install --no-cache-dir -r requirements.txt

# COPY the app code AFTER installing dependencies
COPY app/ ./app/
COPY tests/ ./tests/

# EXPOSE — documents which port the app uses
# WARNING: this does NOT publish the port! -p flag does that.
EXPOSE 8000

# Security: create and run as non-root user
# If container is compromised, attacker gets appuser, not root
RUN adduser --disabled-password --gecos '' appuser
USER appuser

# CMD — the command to run when container starts
# Can be overridden with: docker run image DIFFERENT_COMMAND
CMD ["uvicorn", "app.main:app", "--host", "0.0.0.0", "--port", "8000"]
⚠️ Common mistake: EXPOSE does NOT publish ports. You still need docker run -p 8000:8000 to access it from outside the container.

DEPLOYMENT ☸️ k8s/deployment.yaml — Every Line Explained 
apiVersion: apps/v1       # API version — use apps/v1 for Deployments
kind: Deployment          # What type of object we're creating
metadata:
  name: taskapi           # Name of this Deployment
  namespace: default      # Which namespace (like a folder) it lives in
  labels:
    app: taskapi          # Labels = key:value tags for finding this later
spec:
  replicas: 2             # Run 2 identical Pods at all times
  selector:
    matchLabels:
      app: taskapi        # This Deployment manages Pods with label app=taskapi
  template:               # Template for creating Pods
    metadata:
      labels:
        app: taskapi      # Pods get this label — must match selector above
    spec:
      containers:
        - name: taskapi
          image: yourdockerhubuser/taskapi:latest
          ports:
            - containerPort: 8000
          env:
            - name: APP_ENV
              value: "production"
          resources:
            requests:     # Minimum guaranteed resources
              cpu: "100m" # 100m = 0.1 CPU cores
              memory: "128Mi"
            limits:       # Maximum allowed — pod killed if exceeded
              cpu: "500m"
              memory: "256Mi"
          livenessProbe:  # K8s restarts pod if this fails
            httpGet:
              path: /health
              port: 8000
            initialDelaySeconds: 10  # wait 10s after start before checking
            periodSeconds: 15        # check every 15 seconds
          readinessProbe: # K8s only routes traffic when this passes
            httpGet:
              path: /health
              port: 8000
            initialDelaySeconds: 5
            periodSeconds: 10

CORE CONCEPTS 🏗️ Terraform Mental Model — 5 Concepts to Know
📦 Provider = "which cloud?" — tells Terraform to talk to GCP, AWS, or Azure
🏗️ Resource = "what to create" — a VM, a cluster, a database, a network
🧠 State file = Terraform's memory — records what it created. Never delete this!
📋 Variables = like function parameters — make your code reusable
📤 Outputs = print values after apply — like the cluster IP or connection string
The 4 Terraform commands you'll use 95% of the time:
terraform workflow
$ terraform init # download providers, setup backend
$ terraform plan # PREVIEW what will be created/changed/deleted
$ terraform apply # CREATE the infrastructure (asks for confirmation)
$ terraform destroy # DELETE everything (saves money!)
⚠️ Always run terraform destroy when done learning! A GKE cluster left running costs ~$0.10/hr from your credit.
MAIN.TF 🏗️ main.tf — Create a GKE Cluster (Free with Credit)
Create: mkdir terraform-gke && cd terraform-gke && nano main.tf
# Tell Terraform to use GCP
provider "google" {
  project = var.project_id
  region  = var.region
}

# The Kubernetes cluster
resource "google_container_cluster" "taskapi" {
  name     = "taskapi-cluster"
  location = var.region

  # Remove default node pool — we create our own below
  remove_default_node_pool = true
  initial_node_count       = 1

  networking_mode = "VPC_NATIVE"
  ip_allocation_policy {}
}

# Node pool — the actual VMs that run your pods
resource "google_container_node_pool" "taskapi_nodes" {
  name       = "taskapi-node-pool"
  cluster    = google_container_cluster.taskapi.name
  location   = var.region
  node_count = 2  # 2 VMs in the pool

  node_config {
    machine_type = "e2-standard-2"  # 2 vCPU, 8GB RAM per node
    disk_size_gb = 50
    oauth_scopes = [
      "https://www.googleapis.com/auth/cloud-platform"
    ]
  }

  management {
    auto_repair  = true   # auto-replace unhealthy nodes
    auto_upgrade = true   # auto-update K8s version
  }
}

# Print the connect command after cluster is created
output "connect_command" {
  value = "gcloud container clusters get-credentials taskapi-cluster --region ${var.region} --project ${var.project_id}"
}

PIPELINE ⚙️ .github/workflows/ci.yml — Complete Production Pipeline 
name: TaskAPI CI/CD Pipeline

on:
  push:
    branches: [main, staging]      # trigger on push to these branches
  pull_request:
    branches: [main]               # also runs on PRs to main

env:
  IMAGE: ${{ secrets.DOCKERHUB_USERNAME }}/taskapi

jobs:

  ## ── JOB 1: Run Tests ──────────────────────────
  test:
    name: 🧪 Test
    runs-on: ubuntu-latest         # free GitHub-hosted runner
    steps:
      - uses: actions/checkout@v4  # pull your code into the runner

      - uses: actions/setup-python@v5
        with:
          python-version: "3.11"

      - name: Install dependencies
        run: pip install -r requirements.txt

      - name: Run pytest
        run: pytest tests/ -v --tb=short

  ## ── JOB 2: Build + Push Docker Image ──────────
  build:
    name: 🐳 Build & Push
    needs: test                    # only runs if tests PASS
    runs-on: ubuntu-latest
    if: github.ref == 'refs/heads/main'  # only on main branch
    steps:
      - uses: actions/checkout@v4

      - uses: docker/login-action@v3
        with:
          username: ${{ secrets.DOCKERHUB_USERNAME }}
          password: ${{ secrets.DOCKERHUB_TOKEN }}

      - uses: docker/build-push-action@v5
        with:
          push: true
          tags: |
            ${{ env.IMAGE }}:latest
            ${{ env.IMAGE }}:${{ github.sha }}  # tag with commit hash

  ## ── JOB 3: Deploy to Kubernetes ───────────────
  deploy:
    name: 🚀 Deploy
    needs: build
    runs-on: ubuntu-latest
    if: github.ref == 'refs/heads/main'
    steps:
      - uses: actions/checkout@v4

      - uses: azure/setup-kubectl@v3

      - name: Configure kubectl
        run: echo "${{ secrets.KUBECONFIG }}" | base64 -d > kubeconfig.yaml

      - name: Update image tag
        run: |
          sed -i "s|:latest|:${{ github.sha }}|g" k8s/deployment.yaml

      - name: Deploy
        run: |
          kubectl --kubeconfig=kubeconfig.yaml apply -f k8s/
          kubectl --kubeconfig=kubeconfig.yaml rollout status deployment/taskapi --timeout=120s

      - name: Verify deployment
        run: |
          kubectl --kubeconfig=kubeconfig.yaml get pods
          echo "✅ Deployment complete!"
✓ Flow: git push main → tests run (2min) → image built + pushed (3min) → deployed to K8s (1min) → your new code is live.

Error Bible — Fix Any Error in 5 Minutes
The 20 most common errors you'll hit. Exact fix for each. Save hours of googling.
K8S-01CrashLoopBackOff

Pod starts, crashes, K8s restarts it, it crashes again. Usually: app is crashing on startup, missing env variable, or wrong command.

kubectl logs POD_NAME --previous — read the crash log. Fix the app error or missing env var.
K8S-02ImagePullBackOff

K8s can't pull the Docker image. Wrong image name, wrong tag, or repo is private without credentials.

kubectl describe pod POD_NAME → check Events. Verify image name exactly. For private repos, create K8s imagePullSecret.
K8S-03Pending — no nodes available

Pod is waiting for a node. Either no nodes exist, or nodes don't have enough CPU/memory for your resource requests.

kubectl describe pod → check "Insufficient cpu/memory". Lower your resource requests or add more nodes.
K8S-04OOMKilled — Out of Memory

Container exceeded its memory limit. K8s killed it. App has a memory leak or limit is too low.

Increase memory limit in deployment.yaml OR fix the memory leak in your app. Check kubectl describe pod for "OOMKilled".
DOCKER-01Port already in use

You tried to run container on port 8000 but something already uses it on the host.

lsof -i :8000 to find what's using it. Kill it with kill -9 PID. Or use a different host port: docker run -p 8001:8000
DOCKER-02Cannot connect to Docker daemon

Docker Desktop isn't running, or current user isn't in the docker group on Linux.

Start Docker Desktop. On Linux: sudo usermod -aG docker $USER then log out and back in (or run: newgrp docker).
DOCKER-03Module not found in container

Your requirements.txt is missing a package, or COPY order in Dockerfile is wrong.

Add missing package to requirements.txt. Make sure COPY requirements.txt . is BEFORE RUN pip install.
TF-01Error: no valid credential sources

Terraform can't authenticate to GCP. gcloud CLI isn't logged in or Application Default Credentials missing.

Run: gcloud auth application-default login — opens browser, sign in with your Google account. Re-run terraform apply.
TF-02Error acquiring state lock

Another Terraform process is running, or a previous run crashed and left a lock behind.

terraform force-unlock LOCK_ID (ID shown in error message). Only do this if you're sure no other process is running.
CICD-01Pipeline fails: Permission denied (publickey)

GitHub Actions can't SSH into your server or K8s cluster because the SSH key or kubeconfig secret is wrong.

Re-check your GitHub Secrets. KUBECONFIG must be base64-encoded: cat ~/.kube/config | base64 -w0 and paste as the secret value.
CICD-02Docker push: unauthorized

GitHub Actions can't push to Docker Hub. Token expired or wrong username/token in secrets.

Generate a new Docker Hub Access Token (hub.docker.com → Account → Security → New Access Token). Update DOCKERHUB_TOKEN secret in GitHub.
GIT-01Merge conflict

Two people (or branches) changed the same line in a file. Git doesn't know which version to keep.

Open the conflicted file. Find <<<<<<< markers. Keep the right code. Delete the markers. git add . then git commit.

Top Interview Questions — With Perfect Answers
Click to reveal. Real questions from real DevOps interviews at top companies.
Q1What is the difference between a Docker image and a container?Easy
Image = the read-only blueprint. Built with docker build. Stored in a registry. Like a class definition in code, or a cake recipe. Container = a running instance of that image. Created with docker run. Has a writable layer on top. Like an object/instance, or a baked cake. You can run 10 containers from 1 image simultaneously, each isolated.
🎯 Say in interview: "Image is immutable, stored in registry. Container is ephemeral, running in memory. I always tag images with commit SHA, not :latest, for production."
Q2Explain Kubernetes architecture — what runs on master vs worker?Medium
Control Plane (Master): API Server (entry point — all kubectl commands hit this), etcd (distributed key-value store — the cluster's database/memory), Scheduler (decides which node to run a new pod on), Controller Manager (runs control loops — ensures desired state matches actual state).

Worker Nodes: kubelet (agent that runs on every node, talks to API server, manages pods), kube-proxy (handles network routing to pods), Container Runtime (Docker/containerd — actually runs the containers).
🎯 Say: "The control plane is the brain. Worker nodes are the muscles. kubectl talks to the API server, which writes to etcd, which controllers watch and act on."
Q3What is the difference between CI and CD? Walk me through a pipeline.Easy
CI (Continuous Integration): Every code push automatically triggers: checkout code → install deps → run tests → build Docker image. If anything fails, pipeline stops. No broken code moves forward.

CD (Continuous Delivery/Deployment): On CI success: push image to registry → update K8s manifests → kubectl apply → rolling update deploys.

Full flow: git push → tests (2 min) → docker build + push (3 min) → deploy to K8s (1 min) → new version live in 90 seconds, zero human steps.
🎯 Say: "CI catches bugs early when they're cheap to fix. CD eliminates manual deployments — every merge to main is automatically production-ready."
Q4What is Terraform state? What happens if it's lost?Medium
Terraform state (terraform.tfstate) is Terraform's memory — it records every resource it created, their IDs, and their current configuration. Without state, Terraform doesn't know what already exists. If you run apply without state, it tries to create everything again = duplicates and errors.

If state is lost: you run terraform import to re-associate existing resources with your config, or you manually delete and recreate infrastructure.

Best practice: store state remotely in GCS/S3 with state locking (via Terraform Cloud or backend config) so the team shares it and concurrent runs don't corrupt it.
🎯 Say: "I always use remote state with GCS backend + state locking. Never commit tfstate to git — it may contain secrets."
Q5Pod is in CrashLoopBackOff. How do you debug it?Medium
Step-by-step debug flow:

1. kubectl logs POD_NAME --previous — get logs from the crashed instance (not the restarting one). 90% of causes are here.

2. kubectl describe pod POD_NAME — scroll to Events. Shows: image pull failures, OOMKilled, scheduling issues, probe failures.

3. Check env vars — missing DATABASE_URL or wrong secret name causes startup crash.

4. kubectl exec -it POD_NAME -- sh if you can catch it in a running state, then check connectivity: curl DB_HOST:5432.

5. Temporarily add command: ["sleep", "infinity"] to pod spec to keep it alive and debug inside.
🎯 Say: "My first instinct is --previous logs. CrashLoopBackOff almost always means the app itself is crashing — it's a K8s symptom, not the root cause."
Q6What's the difference between liveness and readiness probes?Easy
Liveness probe: "Is this container alive?" K8s polls /health every N seconds. If it fails K consecutive times, K8s kills and restarts the container. Solves: deadlocks, infinite loops, zombie processes.

Readiness probe: "Is this container ready to serve traffic?" If it fails, the pod is removed from the Service's endpoint list — it won't receive requests, but it ISN'T restarted. Solves: app taking 30s to warm up, app temporarily overloaded.

Both use the same syntax but have opposite consequences: liveness triggers restart, readiness removes from load balancer.
🎯 Mnemonic: Liveness = Launch new pod. Readiness = Remove from rotation.
Q7How do you do a zero-downtime deployment in Kubernetes?Medium
K8s does zero-downtime rolling updates by default. In your deployment, strategy: type: RollingUpdate with maxUnavailable: 0 and maxSurge: 1. This means: bring up 1 new pod, wait for it to pass readiness probe, then kill 1 old pod. Repeat until all pods are updated. At no point is the service down.

Prerequisites for true zero-downtime: readiness probe must pass before traffic is routed, your app must handle graceful shutdown (SIGTERM signal), database schema changes must be backward-compatible (never drop a column in the same deploy that removes code using it).
🎯 Say: "I set maxUnavailable: 0 and rely on readiness probes. I also ensure my app handles SIGTERM gracefully — drains in-flight requests before shutting down."
Q8What is Infrastructure as Code? Why use it over clicking in the console?Easy
IaC = describing your infrastructure in code files (Terraform HCL, Pulumi TS, AWS CDK). Benefits over clicking:

Reproducibility: Create identical staging + prod environments from same code.
Version control: Every infra change is a git commit — you know who changed what and when.
Peer review: Infrastructure changes go through code review like application code.
Disaster recovery: If prod dies, run terraform apply and rebuild in 5 minutes.
Documentation: The code IS the documentation of what exists.

Clicking in a console creates snowflake infrastructure — unique, undocumented, impossible to reproduce exactly.
🎯 Say: "IaC makes infrastructure boring — repeatable, reviewable, testable. Manual clicks make it a snowflake that only one person understands."
Q9How do you monitor application health? What metrics matter?Medium
The four golden signals (from Google SRE book):

Latency: How long requests take. Track p50, p95, p99 — not just average.
Traffic: Requests per second. Baseline + spike detection.
Errors: HTTP 5xx error rate. Alert if >1% of requests fail.
Saturation: How full is the system — CPU%, memory%, queue depth.

Implementation: Prometheus scrapes /metrics every 15s. Grafana visualises. AlertManager fires PagerDuty/Slack when thresholds breach. Loki aggregates logs. This gives you full observability: metrics (what happened), logs (why it happened), traces (where in the code it happened).
🎯 Say: "I follow the four golden signals. Latency and error rate are my primary SLIs. I set SLOs like 99.9% of requests under 200ms, and alert before breaching the error budget."
Q10What is a Kubernetes namespace? When do you use multiple?Hard
Namespace = a virtual cluster within a K8s cluster. Resources in different namespaces are isolated — same name can exist in namespace A and B simultaneously.

Use multiple namespaces for: environment isolation (namespace per environment: dev/staging/prod), team isolation (team-a, team-b each own their namespace), resource quotas per team/app, RBAC — grant team-a access to only their namespace.

Common pattern in companies: one K8s cluster, separate namespaces per environment. Or separate clusters for prod vs non-prod (harder to breach isolation between clusters).

System namespaces (never touch): kube-system (K8s core components), kube-public (cluster info), default (where your stuff goes if you don't specify).
🎯 Say: "I use namespaces to enforce separation of concerns. In my setup: default for dev, staging namespace for staging, and RBAC ensures junior devs can't accidentally kubectl delete -n staging."
