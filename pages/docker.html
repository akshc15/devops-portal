Chapter 03 · Containerisation

🐳 Docker
Pack Once.
Run Anywhere.

Docker is the most important skill in modern DevOps. It solves the oldest problem in software — "it works on my machine". After this chapter, you will containerise anything.

// the problem docker solved
The Story — "It Works On My Machine"
😤 The year is 2012. A developer named Alex pushes code to production.

On Alex's laptop: Python 3.9, library version 2.1.4, Ubuntu 18. Works perfectly.
On the server: Python 3.6, library version 1.8.0, CentOS 7. Crashes instantly.

The ops team spends 3 days figuring out why. The answer: the environment is different. This happened every single time anyone deployed anything. Teams had a saying — the saddest words in software: "But it works on my machine!"

Docker launched in 2013 with one radical idea: what if you packaged not just the code, but the entire environment — the OS, runtime, libraries, config — everything — into one sealed unit that runs identically everywhere? A container.

Today Docker has 20 million developers. Every cloud, every CI/CD pipeline, every Kubernetes cluster uses it. If you know one thing in DevOps, make it Docker.
🧒 Explain Like I'm 5

Think about shipping containers on cargo ships. Before containers existed, every ship was loaded differently — boxes, barrels, random shapes. Loading took weeks. Things got damaged. Shipping containers standardised everything — same box, same size, fits any ship, any truck, any crane. Docker does the same for software. Your app goes in a standard container that runs on any laptop, any server, any cloud — the same way, every time.

3 Core Concepts — Image, Container, Registry
Everything in Docker comes down to these three things. Understand them and every command makes sense.
📸
Image
A read-only blueprint — like a recipe or a class in code
Built from a Dockerfile with docker build
Stored in a registry (Docker Hub)
Never changes once built
One image → unlimited containers
📦
Container
A running instance of an image — like an object from a class
Created with docker run
Has its own isolated filesystem, network, processes
Ephemeral — stop it, data is gone (unless you use volumes)
10 containers can run from 1 image simultaneously

Docker Images are Made of Layers

Each instruction in your Dockerfile creates a new layer. Layers are cached — if layer 3 didn't change, Docker reuses it from cache. This makes rebuilds fast.

FROM python:3.11-slimLayer 1 — Base OS
RUN apt-get updateLayer 2 — System packages
COPY requirements.txt + RUN pip installLayer 3 — Dependencies ← cache this!
COPY app/ ./app/Layer 4 — Your code (changes most often)
Container writable layerLayer 5 — Runtime writes (lost on stop)
💡
Order matters in Dockerfile. Put things that change rarely (apt install, pip install) BEFORE things that change often (COPY app code). This maximises cache hits and makes rebuilds much faster.

Dockerfile — Build TaskAPI Image From Scratch
A Dockerfile is a recipe for building your image. Every instruction creates a layer. Every line explained below.
PART 1 Create the TaskAPI App First ›

We need an app to containerise. Create this structure on your VM:

create project structure
$ mkdir -p taskapi/app taskapi/tests
$ cd taskapi
app/main.py — the API we will containerise
# app/main.py
from fastapi import FastAPI, HTTPException
from pydantic import BaseModel
from typing import List
import uuid
app = FastAPI(title="TaskAPI", version="1.0.0")
tasks_db = {}
class Task(BaseModel):
title: str
done: bool = False
@app.get("/health")
def health(): return {"status": "ok", "version": "1.0.0"}
@app.get("/tasks")
def list_tasks(): return list(tasks_db.values())
@app.post("/tasks", status_code=201)
def create_task(task: Task):
tid = str(uuid.uuid4())
tasks_db[tid] = {"id": tid, **task.dict()}
return tasks_db[tid]
@app.delete("/tasks/{task_id}", status_code=204)
def delete_task(task_id: str):
if task_id not in tasks_db: raise HTTPException(404)
del tasks_db[task_id]
requirements.txt
fastapi==0.111.0
uvicorn==0.29.0
pydantic==2.7.0
PART 2 Write the Dockerfile — Every Line Explained ›
Dockerfile — save as "Dockerfile" (no extension)
# FROM — which base image to start from
# python:3.11-slim = Python 3.11 on minimal Debian
# -slim = stripped down, smaller image (~50MB vs 900MB for full)
FROM python:3.11-slim
# WORKDIR — set working directory INSIDE the container
# All following commands run from /app
# If /app doesn't exist, Docker creates it
WORKDIR /app
# COPY requirements BEFORE the app code — CRITICAL for caching
# If requirements.txt didn't change → pip install is SKIPPED on rebuild
# If app code changes → only the last COPY re-runs (super fast rebuilds)
COPY requirements.txt .
# RUN — executes commands during BUILD (not runtime)
# --no-cache-dir = don't store pip download cache in image
# This makes the image smaller
RUN pip install --no-cache-dir -r requirements.txt
# Copy app code (changes most often → last layer)
COPY app/ ./app/
# EXPOSE — documents which port the app uses
# WARNING: this does NOT publish the port to the host!
# You still need -p 8000:8000 in docker run to access from outside
EXPOSE 8000
# SECURITY: never run as root inside containers
# If attacker breaks in, they get "appuser" not root
RUN adduser --disabled-password --gecos '' appuser
USER appuser
# CMD — command to run when container STARTS
# Use array format (exec form) — not string (shell form)
# --host 0.0.0.0 = listen on all interfaces (not just localhost)
# Without 0.0.0.0 you can't reach the app from outside the container!
CMD ["uvicorn", "app.main:app", "--host", "0.0.0.0", "--port", "8000"]
⚠️
Most common beginner mistake: Forgetting --host 0.0.0.0 in the CMD. Your app starts fine but you can't reach it because it only listens on localhost INSIDE the container. Always use 0.0.0.0 in containers.

v1: digest: sha256:abc123... size: 1234

💡
In production, use commit SHA tags instead of :latest. Example: akshc15/taskapi:a3f2c91. This way you always know exactly which version is running. :latest is ambiguous — nobody knows which version "latest" is.

Docker Compose — Your Local Dev Environment
Real apps need multiple services — app + database + cache + queue. Compose runs them all together with one command.
🧒 Explain Like I'm 5

Running one container is like making one dish. But a real meal needs 4 dishes cooking at the same time. Docker Compose is the full recipe book — it says "start the app dish, start the database dish, start the cache dish, make sure they can all talk to each other, all at once".

:8000
built from ./
🐘
db
:5432
postgres:15
🔴
redis

docker-compose.yml — complete annotated example
version: "3.9"
services:
# ── App Service ─────────────────────────────────────
app:
build: . # build from Dockerfile in current dir
container_name: taskapi
ports:
- "8000:8000" # host:container
environment:
- APP_ENV=development
- DATABASE_URL=postgresql://taskuser:taskpass@db:5432/taskdb
# ↑ "db" = service name, not localhost!
- REDIS_URL=redis://redis:6379
volumes:
- ./app:/app/app # live reload — code changes reflect instantly
depends_on:
db:
condition: service_healthy # wait for db to be ready
restart: unless-stopped
# ── Database Service ─────────────────────────────────
db:
image: postgres:15-alpine # pull from Docker Hub, don't build
container_name: postgres
environment:
POSTGRES_USER: taskuser
POSTGRES_PASSWORD: taskpass
POSTGRES_DB: taskdb
volumes:
- pg_data:/var/lib/postgresql/data # persist data!
healthcheck:
test: ["CMD-SHELL", "pg_isready -U taskuser"]
interval: 10s
timeout: 5s
retries: 5
# ── Redis Cache ──────────────────────────────────────
redis:
image: redis:7-alpine
container_name: redis
volumes:
pg_data: # named volume — data persists between restarts

# Start everything in background
$ docker-compose up -d

Creating redis ... done
Creating taskapi ... done
# See all services status
$ docker-compose ps

$ docker-compose up -d --build
# Stop everything (keeps data in volumes)
$ docker-compose stop
# Stop AND delete containers + networks (keeps volumes)
$ docker-compose down
# Stop AND delete EVERYTHING including volumes (data gone!)
$ docker-compose down -v

Common Docker Errors — Exact Fix
ERR 01 port is already allocated ›
Error: bind: address already in use. Port 8000 is already allocated
# Something else is using port 8000 on your host
$ lsof -i :8000 # find what's using it
$ docker ps # maybe another container
$ docker stop old-container # stop the old one
# Or use a different host port:
$ docker run -p 8001:8000 taskapi # map to 8001 instead
ERR 02 Container starts then immediately exits ›
$ docker ps ← container not showing
$ docker ps -a ← shows it as Exited(1)
taskapi Exited (1) 3 seconds ago
# Check the logs to see why it crashed
$ docker logs taskapi
ModuleNotFoundError: No module named 'fastapi'
# FIX: requirements.txt missing or not copied in Dockerfile
ERR 03 Cannot connect to Docker daemon ›
$ docker ps
Cannot connect to the Docker daemon. Is the docker daemon running?
# FIX on Linux:
$ sudo systemctl start docker
$ sudo systemctl enable docker # start on boot
# Also check if you're in the docker group:
$ groups ← should show "docker" in the list
$ newgrp docker ← apply group without logout
ERR 04 App running but can't reach it in browser ›
$ docker ps ← shows container running
$ curl localhost:8000 ← works from VM
# But browser can't reach it
# FIX: GCP firewall rule needs to allow port 8000
# GCP Console → VPC Network → Firewall → Create rule
# OR: check app listens on 0.0.0.0 not 127.0.0.1
# Wrong: uvicorn app.main:app --host 127.0.0.1
# Right: uvicorn app.main:app --host 0.0.0.0

Docker Interview Questions — Perfect Answers
Q1What is the difference between a VM and a Docker container?Easy
A VM (Virtual Machine) virtualises the entire hardware — it has its own OS kernel, its own memory, its own disk. Boots in minutes. Heavy — typically 1–5GB. Good isolation.

A Docker container shares the host OS kernel. It only packages the app and its dependencies — not the kernel. Starts in milliseconds. Lightweight — typically 50–500MB. Uses less RAM.

Analogy: VM = buying a whole house. Container = renting a room in a shared house. Same security for the room but sharing the foundation.
🎯 Say: "I use containers for application packaging and deployment because they're lightweight and start fast. VMs are useful when you need full OS isolation or a different kernel."
Q2What is a multi-stage Docker build and why use it?Medium
Problem: Build tools (compilers, npm, test frameworks) are needed during build but not at runtime. Without multi-stage builds, your production image contains all build tools = bloated, insecure.

Solution: Multi-stage build uses multiple FROM statements. Stage 1 (builder) installs everything and compiles. Stage 2 (runtime) copies ONLY the final artifact — no build tools included.

Result: Image 5–10× smaller. Fewer packages = fewer vulnerabilities. Faster pulls, faster K8s pod starts.
🎯 Say: "I use multi-stage builds for all production images. For our Python app, the builder stage installs dev dependencies and runs tests. The runtime stage copies only the app code and prod dependencies — image went from 890MB to 85MB."
Q3How do containers communicate with each other in Docker?Medium
By default, each container has its own network namespace — completely isolated. There are 3 ways to connect them:

1. Docker networks (recommended): Create a shared network. Containers on the same network reach each other by service name. In Docker Compose, this is automatic — all services share a network. Your app connects to postgres via db:5432 not localhost:5432.

2. Port mapping: -p 8000:8000 exposes a container port to the host. Containers on different networks reach each other via host IP + port.

3. --network host: Container shares host network. No isolation. Use only in specific cases.
🎯 Say: "In Docker Compose, every service is on the same default network automatically. I reference other services by their service name — db, redis — not localhost. This is a classic interview trap question."
Q4What happens to data when a Docker container is deleted?Easy
Containers are ephemeral by default. The writable layer (everything written during runtime) is destroyed when the container is removed with docker rm. Any data written inside the container — database records, uploaded files, logs — is permanently gone.

Solution: Volumes. A named volume (docker run -v mydata:/var/lib/postgresql/data) stores data on the host filesystem outside the container lifecycle. When you delete and recreate the container, it reattaches to the same volume — data survives.

Bind mounts (-v /host/path:/container/path) mount a host directory into the container — useful for development (live code reload).
🎯 Say: "I always use named volumes for databases and user-uploaded files. For development, I use bind mounts so code changes reflect instantly without rebuilding."
Q5How do you reduce Docker image size?Medium
5 techniques in order of impact:

1. Use slim/alpine base images: python:3.11-slim (50MB) vs python:3.11 (900MB). Alpine is even smaller but uses musl libc — can cause issues with some packages.
2. Multi-stage builds: Don't include build tools in final image.
3. Layer ordering + cache: Copy requirements before code — cache pip install layer.
4. Combine RUN commands: Each RUN = 1 layer. RUN apt-get update && apt-get install -y pkg && rm -rf /var/lib/apt/lists/* in one command + cleanup in same layer.
5. .dockerignore file: Exclude .git, tests, docs, node_modules from the build context.
🎯 Say: "My standard approach: slim base → multi-stage → combine RUN commands → .dockerignore → verify with docker history. Our TaskAPI image is under 130MB."
