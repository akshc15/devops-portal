🐧 Linux
From Zero to Confident

Before Docker, Kubernetes, Terraform — before any DevOps tool — there is Linux. Every server you will ever touch runs Linux. This is where everything begins.

// why linux exists
The Story — Why Linux Was Born
📺 1991. A 21-year-old Finnish student named Linus Torvalds was frustrated.

The best operating systems at the time — Unix and Windows — cost thousands of dollars. Students and hobbyists couldn't afford them. Linus wanted a free, open, powerful OS he could study, modify, and share.

So he built one. In his bedroom. He posted it online with a message: "I'm doing a free operating system — just a hobby, won't be big and professional like GNU."

That "hobby" now runs 96% of the world's servers, every Android phone, every supercomputer, every cloud platform — AWS, GCP, Azure all run Linux underneath. Your Docker containers run Linux. Your Kubernetes nodes run Linux. Even when you use a Mac, the commands you'll learn here are the same ones professionals use every day.
🧒 Explain Like I'm 5

Imagine your computer is a house. Windows is like renting a fancy apartment — everything is decided for you, you can't change the walls, and you pay monthly. Linux is like owning an empty plot of land — you build exactly what you want, nothing is hidden, and it's completely free. DevOps engineers live on that plot of land because they need total control over their servers.

What Linux Actually Is
Three things you must understand before touching a command.
🧠
The Kernel
The core of Linux. It's the software that talks directly to hardware — CPU, RAM, storage. Like the engine of a car. You never touch it directly.
🐚
The Shell
The shell is how you talk TO Linux by typing commands. Bash is the most common shell. When you open a terminal and type, you're talking to the shell. The shell sends your request to the kernel.
📦
The Distro
Linux comes in flavours called distributions (distros). Ubuntu, Debian, CentOS, Alpine. We use Ubuntu 22.04 — it's the most popular for DevOps. All distros use the same core commands.
🪟
Windows
Great for desktop. Expensive licences. Limited for servers. Most cloud servers don't run it.
🐧
Linux (Ubuntu)
Free. Runs 96% of servers. What AWS, GCP, Azure use. What Docker, K8s run on. Learn this.
🍎
macOS
Unix-based (like Linux). Terminal commands are similar. Good for development but too expensive for servers.
💡
You don't install Linux on your laptop. You create a free cloud VM on GCP that runs Linux, and you connect to it from your Mac/Windows. Your laptop stays untouched. All commands run safely in the cloud.

Tilde
Shortcut for your home directory. /home/yourname. You always start here.

The Linux File System
Linux organises everything as files in a tree. No drive letters like C:\ or D:\. Everything starts from /
🧒 Explain Like I'm 5

In Windows you have C:\ and D:\. In Linux there's only one tree, starting from / (called "root"). Think of it like a building — the basement is /, and every room is a folder inside it. Your stuff lives in /home/yourname — that's your flat in the building.

/ ← the ROOT of everything (like C:\ in Windows)
├── home/ ← user home directories
│ └── akshay/ ← YOUR home folder (~)
│ ├── projects/ ← you create this
│ └── notes.txt
├── etc/ ← system config files (nginx.conf, ssh config live here)
├── var/ ← logs and variable data (var/log/syslog)
├── usr/ ← installed programs (usr/bin/python3)
├── bin/ ← essential commands (ls, cat, echo live here)
├── tmp/ ← temporary files, deleted on reboot
└── proc/ ← running processes (virtual, not real files)
💡
The most important folder for you: ~ = /home/yourname. Always work here. Never run random commands in /etc or / unless you know what you're doing.

The pipe | is one of Linux's superpowers. It chains commands — output of the left command becomes input of the right. cat log | grep ERROR | wc -l counts how many errors in a log file. One line. Powerful.

File Permissions — Demystified
Every file has 3 permission groups. Once you understand this, it never confuses you again.
🧒 Explain Like I'm 5

Imagine a notebook in an office. The owner (you) can read and write it. Your team can only read it. Visitors (strangers) can't touch it at all. Linux does the same with every file — it assigns read (r), write (w), execute (x) permissions for 3 groups: owner, group, others.

reading permissions
$ ls -la
-rwxr-xr-- akshay staff deploy.sh
│ ││││ │││ │
│ ││││ │││ └── filename
│ rwx = OWNER can Read+Write+Execute
r-x = GROUP can Read+Execute (no write)
r-- = OTHERS can only Read
Owner (you)
rwx
Group (team)
r-x
Others (world)
r--
chmod — change permissions
# Make a script executable (you'll do this constantly)
$ chmod +x deploy.sh # add execute permission
$ ./deploy.sh # now you can run it

# Numeric method (very common)
# 4=read 2=write 1=execute (add them up)
# 7=rwx 6=rw- 5=r-x 4=r--
$ chmod 755 deploy.sh # owner=rwx group=r-x others=r-x
$ chmod 600 secret.key # only owner can read/write (private key)
$ chmod 644 app.py # standard for files

Shell Scripting — Your First Automation
A shell script is just a text file with Linux commands. Instead of typing 10 commands manually, you write them once and run the file.
Why scripting matters in DevOps:

Imagine you deploy your app every day. You run the same 8 commands every time. One day you forget step 5. Production breaks. A shell script means you write those 8 commands once — then you just run one file. Same result, every time, no human error. That's automation. That's DevOps.
deploy.sh — your first real script
#!/bin/bash
# The first line MUST be #!/bin/bash — tells Linux: use bash to run this file
# Lines starting with # are comments — ignored by Linux, read by humans

# set -e = stop script immediately if any command fails
# Without this, script continues even after errors — very dangerous
set -e

# VARIABLES — store values you reuse
APP_NAME="taskapi" # no spaces around =
VERSION="1.0.0"
PORT=8000

# echo = print something (like print() in Python)
echo "Deploying $APP_NAME version $VERSION..."

# FUNCTIONS — reusable blocks (define first, call later)
check_requirements() {
echo "Checking requirements..."
if ! command -v docker &> /dev/null; then
echo "ERROR: Docker not installed!"
exit 1 # exit with error code 1
fi
echo "Docker found: $(docker --version)"
}

# IF statement
deploy() {
if [ "$1" == "production" ]; then # $1 = first argument passed to script
echo "Deploying to PRODUCTION"
else
echo "Deploying to STAGING"
fi

# FOR loop — iterate over items
for service in app db redis; do
echo "Starting $service..."
docker start $service 2>/dev/null || echo "$service already running"
done
}

# MAIN — call functions in order
check_requirements
deploy "$1" # pass first argument to deploy function
echo "Done! App running on port $PORT"
run the script
$ chmod +x deploy.sh # make executable (only needed once)
$ ./deploy.sh production # run with "production" as argument
Deploying taskapi version 1.0.0...
Checking requirements...
Docker found: Docker version 25.0.4
Deploying to PRODUCTION
Starting app...
Starting db...
Done! App running on port 8000

Processes — Running and Managing Them
process management
# Run process in BACKGROUND with &
$ python3 app.py & # starts, returns prompt immediately
[1] 12345 # job number + process ID (PID)

# Find a process ID
$ ps aux | grep python
akshay 12345 python3 app.py

$ kill -9 12345 # force stop (SIGKILL) — if normal kill fails

$ lsof -i :8000
COMMAND PID USER TYPE NODE NAME

Linux Interview Questions — With Perfect Answers
Click any question to reveal the answer + what to say in your interview.
Q1What is the difference between a process and a thread?Medium
A process is an independent program running in memory with its own memory space — like a Docker container. A thread is a lightweight unit of execution within a process — multiple threads share the same memory of their parent process. Processes are isolated (crash of one doesn't affect others). Threads are faster to create but share memory (one bad thread can corrupt others). In DevOps: each Docker container is typically one process. When your app handles 1000 concurrent requests, it uses threads (or async I/O) internally.
🎯 Say: "Process = isolated worker with own memory. Thread = sub-worker sharing memory. I always check running processes with ps aux and htop when debugging performance on Linux servers."
Q2What does chmod 755 mean?Easy
chmod 755 sets permissions using octal notation: 7 (rwx) for owner — full read+write+execute. 5 (r-x) for group — read and execute, no write. 5 (r-x) for others — read and execute, no write. In practice: scripts and executables get 755 (everyone can run it, only owner can modify). Private files like SSH keys get 600 (only owner reads/writes, nobody else can touch).
🎯 Say: "I use 755 for scripts, 644 for regular files, 600 for private keys. chmod +x is the quickest way to make something executable."
Q3How do you find which process is using port 8080?Easy
Two ways: lsof -i :8080 — lists open files, filtered to port 8080. Shows the process name, PID, and user. ss -tlnp | grep 8080 — socket statistics, faster than netstat. Once you have the PID, use kill -9 PID to stop it, or kill PID for a graceful stop.
🎯 Say: "lsof -i :PORT is my first command when I get 'address already in use' errors. Then kill -9 the conflicting process."
Q4What is the difference between soft link and hard link?Medium
A hard link creates another name pointing to the same data on disk. If you delete the original file, the hard link still works — data only deleted when all hard links are removed. A soft link (symlink) is like a shortcut — points to the file path, not the data. If original is deleted, symlink breaks. In DevOps: symlinks are used everywhere — /usr/bin/python → /usr/bin/python3.11. Configuration files often symlinked from /etc/nginx/sites-enabled/ to sites-available/.
🎯 Say: "ln -s source dest creates a symlink. I use symlinks for config management and to switch between versions without changing every script that references a path."
Q5How do you check disk space and what do you do when it's full?Easy
df -h shows disk usage per filesystem (h = human readable GB/MB). du -sh /* shows which directories are largest. When disk is full: first run df -h to confirm, then du -sh /var/log/* to find big log files (biggest culprit), then docker system prune to clean up old Docker images/containers, then journalctl --vacuum-size=100M to trim system logs.
🎯 Say: "In production I set up Prometheus alerts at 80% disk usage so I know before it hits 100%. df -h + du -sh are my first diagnostic commands."
