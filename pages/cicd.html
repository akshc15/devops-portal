The Story — Deployment Friday
😱 "Deployment Friday" — the most feared day in software teams before CI/CD.

A developer writes code all week. On Friday afternoon, they SSH into the production server, copy files manually, restart the app and pray. Something breaks. It's 5pm Friday. Nobody wants to stay late to fix it. The site is down all weekend.

Teams were so scared of deploying they did it once a month. Batching up 4 weeks of changes into one massive deployment that was guaranteed to break something.

CI/CD inverted this entirely. Deploy many small changes automatically. Each one tested. Each one reversible. Netflix deploys thousands of times per day. Amazon deploys every 11 seconds. Small deployments are safe deployments. Automation removes human error.
🧒 Explain Like I'm 5

Imagine a factory that makes toys. Old way: workers build toys all month, then test them all at once, then ship them — one giant stressful day. CI/CD way: every toy gets automatically tested on the assembly line right after it's made. If it passes — it ships immediately. If it fails — the line stops and fixes it right there. Small batches. Constant flow. No surprises.

① Trigger
Developer pushes to GitHub. Pipeline starts automatically in <5 seconds.
② Test
Run all unit tests + linting. If any test fails → pipeline stops. Bad code never moves forward.
③ Build
docker build — creates a new image tagged with the commit SHA. Immutable artifact.
④ Push
docker push — uploads image to Docker Hub / ECR. Kubernetes will pull from here.
⑤ Deploy
kubectl apply — updates K8s deployment with new image. Rolling update. Zero downtime.
⑥ Verify
Check deployment status. Run smoke tests against live URL. Alert on failure.

GitHub Actions — Core Concepts
Workflow
A YAML file in .github/workflows/. Defines what runs, when it runs, and how. One repo can have multiple workflows.
Event / Trigger
What starts the workflow. push, pull_request, schedule (cron), workflow_dispatch (manual button), release.
Job
A unit of work that runs on a fresh virtual machine (runner). Jobs run in parallel by default. Use needs: to sequence them.
Step
An individual task within a job. Either a shell command (run:) or a pre-built action (uses:). Steps run sequentially.
Runner
The virtual machine that runs your job. GitHub provides free Ubuntu, Windows, Mac runners. ubuntu-latest is most common. 2,000 free minutes/month.
Action
A pre-built, reusable step from the GitHub Marketplace. actions/checkout@v4 checks out code. docker/build-push-action builds and pushes Docker images.
Common Triggers
push
On git push to specified branches. Most common trigger.
pull_request
On PR opened/updated. Run tests before merging.
schedule
Cron schedule. Run nightly tests, weekly backups.
workflow_dispatch
Manual trigger. Adds a "Run workflow" button in GitHub UI.
release
On GitHub release published. Deploy tagged versions.

GitHub Actions YAML Syntax — Every Part Explained
.github/workflows/ci.yml — anatomy
# Workflow name — shows in GitHub Actions tab
name: CI Pipeline
# WHEN does this run?
on:
push:
branches: [main, develop] # only on these branches
pull_request:
branches: [main]
# Environment variables available to ALL jobs
env:
IMAGE: ${{ secrets.DOCKERHUB_USERNAME }}/taskapi
jobs:
# ── JOB 1: Test ──────────────────────────────────────
test:
name: 🧪 Run Tests
runs-on: ubuntu-latest # free GitHub-hosted runner
steps:
# Step 1: Get the code (ALWAYS first step)
- name: Checkout code
uses: actions/checkout@v4
# Step 2: Set up Python
- name: Set up Python
uses: actions/setup-python@v5
with:
python-version: "3.11"
# Step 3: Install dependencies
- name: Install dependencies
run: pip install -r requirements.txt
# Step 4: Run tests
- name: Run pytest
run: pytest tests/ -v --tb=short
# ── JOB 2: Build + Push Docker ───────────────────────
build:
name: 🐳 Build & Push
needs: test # only runs if test job PASSES
runs-on: ubuntu-latest
if: github.ref == 'refs/heads/main' # only on main branch
steps:
- uses: actions/checkout@v4
# Login to Docker Hub using secrets
- uses: docker/login-action@v3
with:
username: ${{ secrets.DOCKERHUB_USERNAME }}
password: ${{ secrets.DOCKERHUB_TOKEN }}
# Build and push with commit SHA tag
- uses: docker/build-push-action@v5
with:
push: true
tags: |
${{ env.IMAGE }}:latest
${{ env.IMAGE }}:${{ github.sha }} # exact commit
# ── JOB 3: Deploy to Kubernetes ──────────────────────
deploy:
name: 🚀 Deploy
needs: build
runs-on: ubuntu-latest
if: github.ref == 'refs/heads/main'
steps:
- uses: actions/checkout@v4
- uses: azure/setup-kubectl@v3
# Decode and write kubeconfig from secret
- name: Configure kubectl
run: |
echo "${{ secrets.KUBECONFIG }}" | base64 -d > kubeconfig.yaml
# Update image tag in deployment.yaml with commit SHA
- name: Update image tag
run: |
sed -i "s|:latest|:${{ github.sha }}|g" k8s/deployment.yaml
- name: Deploy to K8s
run: |
kubectl --kubeconfig=kubeconfig.yaml apply -f k8s/
kubectl --kubeconfig=kubeconfig.yaml rollout status deployment/taskapi --timeout=120s
// keep credentials safe
GitHub Secrets — Store Credentials Safely
Never put passwords, tokens, or keys in your YAML files. GitHub Secrets encrypts them and injects at runtime.
STEP 1Add Docker Hub Secrets›
🖱️
Get Docker Hub Token: hub.docker.com → click avatar → Account Settings → Security → New Access Token → name it "github-actions" → Generate → copy the token

Add to GitHub: Your repo → Settings → Secrets and variables → Actions → New repository secret:
Name: DOCKERHUB_USERNAME → Value: your Docker Hub username
Name: DOCKERHUB_TOKEN → Value: the token you copied
STEP 2Add Kubernetes Cluster Secret›
get kubeconfig and encode it
# Get kubeconfig for GKE cluster
$ gcloud container clusters get-credentials taskapi-cluster --region us-central1
# Base64 encode it (pipe to clipboard or copy output)
$ cat ~/.kube/config | base64 -w 0
LS0tLS1CRUdJTiBDRVJUSUZJQ0FURS0t... ← copy this entire output
🖱️
GitHub repo → Settings → Secrets → New secret → Name: KUBECONFIG → Value: paste the base64 output → Add secret
using secrets in workflows
# Access secret with ${{ secrets.SECRET_NAME }}
- name: Login to Docker Hub
uses: docker/login-action@v3
with:
username: ${{ secrets.DOCKERHUB_USERNAME }}
password: ${{ secrets.DOCKERHUB_TOKEN }}
# GitHub also provides built-in variables:
${{ github.sha }} ← commit SHA: a3f2c91...
${{ github.ref }} ← branch: refs/heads/main
${{ github.actor }} ← who triggered: akshc15
${{ github.repository }} ← akshc15/taskapi

Advanced Patterns — Used in Real Companies
PATTERN 1Dependency Caching — Make Pipelines 3x Faster›

Without caching, pip install runs on every push — slow. With caching, dependencies are reused if requirements.txt didn't change.

cache pip dependencies
- name: Cache pip
uses: actions/cache@v4
with:
path: ~/.cache/pip
key: ${{ runner.os }}-pip-${{ hashFiles('requirements.txt') }}
# key changes only if requirements.txt changes → cache miss
restore-keys: ${{ runner.os }}-pip-
- name: Install dependencies
run: pip install -r requirements.txt
# Uses cached packages if key matches → seconds instead of minutes
PATTERN 2Matrix Builds — Test on Multiple Versions at Once›
matrix strategy
test:
runs-on: ubuntu-latest
strategy:
matrix:
python-version: ["3.10", "3.11", "3.12"]
# Creates 3 parallel jobs, one per Python version
steps:
- uses: actions/setup-python@v5
with:
python-version: ${{ matrix.python-version }}
PATTERN 3Manual Approval — Required for Production Deploy›

For production, you may want a human to approve before deployment. GitHub Environments with protection rules add a pause.

🖱️
Repo → Settings → Environments → New environment → name it production → tick Required reviewers → add yourself → Save. Now any job targeting environment: production will pause and wait for your approval in the GitHub UI.
production environment with approval
deploy-prod:
name: 🚀 Deploy Production
needs: build
environment: production ← pauses for approval
runs-on: ubuntu-latest
steps:
- name: Deploy to prod K8s
run: kubectl apply -f k8s/prod/

CI/CD Interview Questions
Q1What is the difference between Continuous Integration, Delivery, and Deployment?Easy
Continuous Integration (CI): Every code push automatically triggers build + test. Goal: catch bugs early when they're cheap. If tests fail, pipeline stops. Developers get feedback in minutes, not days.

Continuous Delivery (CD): Every passing build is automatically prepared for release to production. The release is automatic up to a staging environment but requires a human to click a button to go to production. Every commit is READY to deploy, but not automatically deployed.

Continuous Deployment: Every passing build is automatically deployed to production with no human approval. Netflix, Amazon use this. Requires excellent test coverage and monitoring.
🎯 Say: "We use Continuous Delivery — automatic to staging, manual approval to production. This gives us deployment confidence without risk. We're moving toward Continuous Deployment as our test coverage improves."
Q2How do you handle secrets in CI/CD pipelines?Medium
Never hardcode secrets in workflow YAML files or source code. Three approaches in order of security:

1. GitHub Secrets (good): Stored encrypted in GitHub, injected as environment variables at runtime. Masked in logs. Good for most use cases.

2. Cloud-native secrets (better): AWS Secrets Manager, GCP Secret Manager. Secrets stored in cloud, retrieved at runtime. Rotation support. Audit trail.

3. HashiCorp Vault (best): Dedicated secrets management. Dynamic secrets (short-lived credentials), full audit log, fine-grained access policies. Overkill for small teams but standard at large companies.
🎯 Say: "I use GitHub Secrets for Docker Hub credentials and kubeconfig. For production, I'm moving toward GCP Secret Manager to get rotation and audit trails. The principle: secrets never touch source code or logs."
Q3What is GitOps and how does it differ from traditional CI/CD?Hard
Traditional CI/CD: Pipeline actively pushes changes to infrastructure — kubectl apply, terraform apply run inside the pipeline. Pipeline has credentials to modify production.

GitOps: Git is the single source of truth. An agent running INSIDE the cluster (ArgoCD, Flux) watches a Git repo. When it detects a change, it pulls and applies. The pipeline never has production credentials — it only pushes to git. The cluster pulls changes.

Benefits: security (no outbound credentials from cluster), auditability (every change is a git commit), drift detection (agent continuously reconciles), disaster recovery (recreate cluster from git repo).
🎯 Say: "GitOps inverts the deployment model — instead of pipeline pushing to cluster, cluster pulls from git. I've implemented ArgoCD watching our k8s-manifests repo. Rollback is a git revert. Drift is auto-corrected."
