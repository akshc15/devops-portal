Stop clicking in cloud consoles. Write code that creates servers, clusters, networks, databases — automatically, repeatably, reviewably. This is Infrastructure as Code.

The Story — Clicking is Not Reproducible
🖱️ 2012. A DevOps engineer named Alex spent 3 days clicking through AWS Console to set up an environment.

VPC here. Subnets there. Security groups. EC2 instances. Load balancers. RDS database. IAM roles. 47 different screens. 200+ clicks. It worked.

Six months later, the production environment broke. Alex needed to rebuild staging to match. Could he reproduce those 200 clicks exactly? No. He forgot some settings. The environments drifted. Bugs appeared in production that didn't exist in staging.

HashiCorp launched Terraform in 2014 with one idea: infrastructure should be code. Write a file describing what you want. Terraform creates it. Every time. Identically. Version-controlled. Reviewable. Destroyable with one command.
🧒 Explain Like I'm 5

Clicking in AWS Console is like ordering food by pointing at random items — nobody knows what you ordered, you can't repeat it, and tomorrow you'll forget. Terraform is like writing a recipe — exact ingredients, exact steps, anyone can follow it, and you can make the same dish 100 times perfectly. Your infrastructure becomes a recipe stored in Git.

Core Concepts — 5 Things to Know
Provider
Tells Terraform which cloud to talk to. GCP, AWS, Azure, Cloudflare — each is a provider with its own plugin. You declare it at the top of your config.
Resource
A piece of infrastructure you want to create. A VM, a database, a network, a Kubernetes cluster. Each resource has a type (google_compute_instance) and configuration.
State File
Terraform's memory. Records every resource it created. Used to compare desired vs actual state. Without it, Terraform doesn't know what exists. Never delete it.
Variables
Make your config reusable. Instead of hardcoding "us-central1", use var.region. Change once in variables.tf, updates everywhere.
Outputs
Print values after apply — like the cluster IP, the database URL, the load balancer address. Reference outputs in other configs or just read them.
Module
Reusable Terraform code. Like a function — package a VPC + subnets + firewall rules into one module. Call it 3 times to create 3 environments identically.
terraform init
Download providers + set up backend. Run once per project.
terraform plan
Preview what WILL change. Always run before apply. No changes made yet.
terraform apply
Create/update infrastructure. Asks for confirmation. Changes are made here.
terraform destroy
Delete everything. Save money when done learning. Run this always.
⚠️
Always run terraform plan before terraform apply. Plan shows you exactly what will be created, changed, or destroyed. Surprises in infrastructure cost money and time. Plan is free. Apply is not.

HCL Syntax — HashiCorp Configuration Language
Terraform uses HCL — easy to read, easy to write. Learn these patterns and you can write any Terraform config.
hcl syntax — all the patterns you need
# ── PROVIDER — tell Terraform which cloud ──────────────
terraform {
required_providers {
google = {
source = "hashicorp/google"
version = "~> 5.0" # ~> means "5.x but not 6.0"
}
}
}
provider "google" {
project = var.project_id # uses a variable
region = var.region
}
# ── VARIABLES ──────────────────────────────────────────
variable "project_id" {
description = "GCP project ID"
type = string
}
variable "region" {
default = "us-central1" # default = optional
}
# ── RESOURCE — a thing to create ───────────────────────
resource "google_compute_instance" "vm" {
# type ↑ ↑ local name (used to reference this)
name = "devops-vm"
machine_type = "e2-standard-2"
zone = "${var.region}-a" # string interpolation
boot_disk {
initialize_params { image = "ubuntu-os-cloud/ubuntu-2204-lts" }
}
network_interface { network = "default" }
}
# ── OUTPUT — print values after apply ──────────────────
output "vm_ip" {
value = google_compute_instance.vm.network_interface[0].access_config[0].nat_ip
# resource_type.local_name.attribute
}

Create a Real GKE Cluster with Terraform
This creates a production-grade Kubernetes cluster on GCP using your $300 free credit. Takes ~5 minutes.
MAIN.TF The Complete Terraform Project — Every File ›
project structure
$ mkdir terraform-gke && cd terraform-gke
$ touch main.tf variables.tf outputs.tf terraform.tfvars
main.tf — the GKE cluster
terraform {
required_providers {
google = { source = "hashicorp/google", version = "~> 5.0" }
}
}
provider "google" {
project = var.project_id
region = var.region
}
# Enable required GCP APIs
resource "google_project_service" "container" {
service = "container.googleapis.com"
disable_on_destroy = false
}
# The GKE cluster (control plane)
resource "google_container_cluster" "taskapi" {
name = "taskapi-cluster"
location = var.region
remove_default_node_pool = true # we add our own below
initial_node_count = 1
networking_mode = "VPC_NATIVE"
ip_allocation_policy {}
depends_on = [google_project_service.container]
}
# Node pool — actual VMs that run your pods
resource "google_container_node_pool" "nodes" {
name = "taskapi-nodes"
cluster = google_container_cluster.taskapi.name
location = var.region
node_count = var.node_count
node_config {
machine_type = var.machine_type
disk_size_gb = 50
oauth_scopes = ["https://www.googleapis.com/auth/cloud-platform"]
}
management { auto_repair = true; auto_upgrade = true }
}
variables.tf
variable "project_id" { type = string }
variable "region" { default = "us-central1" }
variable "node_count" { default = 2 }
variable "machine_type" { default = "e2-standard-2" }
terraform.tfvars — your values (gitignored if sensitive)
project_id = "your-gcp-project-id" # find in GCP console top bar
region = "us-central1"
node_count = 2
machine_type = "e2-standard-2"
outputs.tf
output "cluster_name" {
value = google_container_cluster.taskapi.name
}
output "connect_command" {
value = "gcloud container clusters get-credentials ${google_container_cluster.taskapi.name} --region ${var.region} --project ${var.project_id}"
}

State File — The Most Important File
Your Code (main.tf)
resource "gke_cluster" "x"
node_count = 2
machine = e2-standard-2
→ compare →
State File (terraform.tfstate)
gke_cluster: EXISTS
id: projects/x/clusters/y
node_count: 2
remote state — store in GCS (best practice)
# Create GCS bucket for state (do once)
$ gsutil mb gs://YOUR_PROJECT_ID-terraform-state
$ gsutil versioning set on gs://YOUR_PROJECT_ID-terraform-state
# Add backend block to terraform {} in main.tf
terraform {
backend "gcs" {
bucket = "YOUR_PROJECT_ID-terraform-state"
prefix = "taskapi/state"
}
}
# Now state is stored in GCS — safe, shared, versioned
💡
Why remote state? Local state file (terraform.tfstate) is on your laptop — if your laptop dies, Terraform loses track of everything it created. Remote state in GCS is safe, backed up, and multiple team members can share it. State locking prevents two people running apply simultaneously.

ERR 01Error: no valid credential sources›

$ gcloud auth application-default login
ERR 02Error acquiring the state lock›
Error acquiring the state lock: ID: abc-123-def
# Previous run crashed and left a lock
$ terraform force-unlock abc-123-def
# Only if you're certain no other process is running
ERR 03API not enabled›
Error: googleapi: Error 403: Kubernetes Engine API has not been enabled
# FIX: enable the API (or add google_project_service resource)
$ gcloud services enable container.googleapis.com

Terraform Interview Questions
Q1What is Terraform state and what happens if it's lost?Medium
State is Terraform's record of what infrastructure it created. It maps your code resources to real cloud resources (VM ID, IP, etc). Terraform uses state to determine what to change on next apply — if state says a VM exists but your code removes it, Terraform destroys the VM.

If state is lost: Terraform thinks nothing exists. Running apply creates everything again — duplicate resources. Fix: use terraform import to re-associate existing resources with your code, or restore from backup.

Prevention: always use remote state (GCS/S3) with versioning enabled.
🎯 Say: "I always use remote state with GCS backend and versioning. I never commit tfstate to git — it may contain sensitive values. I treat state as the source of truth for what's deployed."
Q2What is the difference between terraform plan and terraform apply?Easy
terraform plan is a dry run — it shows you what WOULD happen (create, update, destroy) without making any changes. Safe to run anytime. Output shows green + (create), yellow ~ (update), red - (destroy).

terraform apply executes the plan — makes actual API calls to create/change/destroy resources. Asks for confirmation unless you use -auto-approve.

Best practice: always run plan first, review output carefully (especially red - lines), then apply. In CI/CD pipelines: plan on PR, apply on merge to main.
🎯 Say: "I run plan on every PR as part of CI — reviewers see exactly what infrastructure will change before it's approved. Apply only runs automatically after merge to main."
Q3What are Terraform modules and why use them?Medium
A module is a reusable package of Terraform code. Like a function — takes inputs (variables), creates resources, returns outputs. Benefits: DRY (don't repeat yourself) — write VPC config once, reuse for dev/staging/prod by passing different variables. Tested, versioned, shareable.

Two types: local modules (in your repo: ./modules/gke-cluster), public modules from Terraform Registry (registry.terraform.io) — e.g., terraform-google-modules/kubernetes-engine/google.

Using a module: module "gke" { source = "./modules/gke-cluster"; project_id = var.project_id }
🎯 Say: "I structure large Terraform projects with modules — one module per service or environment component. It makes adding a new environment a 10-line change instead of 200 lines of copy-paste."
