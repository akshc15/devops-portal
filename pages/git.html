🐙 Git + GitHub
Never Lose Your Work Again

Git is the time machine for your code. GitHub is where you store it online. Every company on the planet uses both. This is not optional — this is the foundation of how software is built.

// why git was invented
The Story — A Problem Every Developer Had
😩 Before Git existed, developers had this nightmare:

You're building a feature. It works perfectly. You add more code. Now it's broken. You try to go back — but you can't. You overwrote the old version. It's gone.

Teams had folders named: project_final.zip, project_final_v2.zip, project_ACTUAL_final.zip, project_use_this_one.zip. Two developers editing the same file at the same time? One person's work gets deleted.

In 2005, Linus Torvalds (same person who built Linux) built Git in 10 days to manage the Linux kernel — which had 1,000+ developers changing code simultaneously. Git solved every one of these problems. Today, 95%+ of software companies use Git. No exceptions.
🧒 Explain Like I'm 5

Imagine you're writing a school essay. Git is like having a magical Save button — but instead of replacing your old save, it keeps every single save forever. You can go back to how the essay looked on Monday. On Tuesday. On any day. You can create a parallel copy of the essay, try crazy changes, and if they're bad — throw that copy away. Your original is completely safe. GitHub is the Google Drive where all these saves are stored online.

4 Concepts — Understand These First
Every Git command makes sense once you understand these 4 ideas.
The 3 Areas Git Manages
💻
Working Directory
Your actual files on disk. Where you write code. Changes here are "unstaged" — git sees them but hasn't saved them yet.
git status
→
📋
Staging Area
A "waiting room" before saving. You choose exactly which changes go into the next save. Like packing a box before shipping.
git add
→
📸
Repository
The permanent history. Every commit is a snapshot saved forever. Never changes. This is your time machine.
git commit
Concept 1
Commit
A snapshot of your files at a point in time. Like a photo. Has a unique ID (hash), author, date, and message. You create one with git commit. Once created, it never changes.
Concept 2
Branch
A parallel version of your code. You branch off main to work on a feature without breaking the original. When done, you merge it back. main is always the stable version.
Concept 3
Remote
A copy of your repo on GitHub's servers. origin is the default name for your GitHub remote. git push sends commits to GitHub. git pull gets changes from GitHub.
Concept 4
Merge / PR
Combining a branch back into main. A Pull Request (PR) is a proposal to merge — teammates review your code before it's merged. Standard in every professional team.

Basic Git Commands — Used Every Single Day
Master these 10 commands and you can handle 90% of real work.
📁

1. Start or clone a project

Either initialise a new repo with git init or get an existing one with git clone URL. Cloning copies the entire history — not just the current files.

✏️

2. Make changes

Edit files normally. Git watches everything. Run git status anytime to see what changed.

📋

3. Stage changes

git add filename or git add . for everything. This puts changes in the "waiting room" before saving.

📸

4. Commit — save a snapshot

git commit -m "what you did". This creates a permanent snapshot. Good messages = your future self thanks you.

🚀

5. Push to GitHub

git push origin main. Sends your local commits to GitHub. Now your work is safe online and visible to teammates.

complete daily workflow — run these in order
# ── START: new project from scratch ────────────────────
$ mkdir taskapi && cd taskapi
$ git init # initialise empty git repo
Initialized empty Git repository in /home/akshay/taskapi/.git/
# ── OR: clone existing project from GitHub ─────────────
$ git clone git@github.com:akshc15/taskapi.git
Cloning into 'taskapi'...
Receiving objects: 100% (42/42), done.
# ── DAILY WORK LOOP ────────────────────────────────────
# 1. See what changed
$ git status
On branch main
Changes not staged for commit:
modified: app.py
modified: Dockerfile
Untracked files:
tests/test_app.py ← new file git hasn't seen
# 2. See exactly what changed inside files
$ git diff # shows line-by-line changes
# 3. Stage ALL changes
$ git add . # stage everything
$ git add app.py # stage only one file
# 4. Commit with a good message
$ git commit -m "Add health endpoint and tests"
[main a3f2c91] Add health endpoint and tests
3 files changed, 47 insertions(+), 2 deletions(-)
# 5. Push to GitHub
$ git push origin main
Enumerating objects: 7, done.
To github.com:akshc15/taskapi.git
b4f1e3d..a3f2c91 main -> main
# 6. View history
$ git log --oneline # compact history
a3f2c91 Add health endpoint and tests
b4f1e3d Add Dockerfile
c8a9d12 Initial commit
💡
Good commit message formula: Start with a verb in present tense. "Add user auth" not "Added user auth". "Fix database timeout" not "Fixed stuff". Keep under 72 characters. Future you will thank current you.

Branches — Parallel Universes for Your Code
main is always stable. You branch off to try things. If it works — merge it. If not — delete the branch. Main never breaks.
🧒 Explain Like I'm 5

Imagine a river (main branch). You dig a side canal (feature branch) to test if fish can live there. You experiment freely. If the fish survive, you connect the canal back to the river (merge). If they don't, you fill the canal in (delete branch) and the main river is completely unaffected.

main ●─────●─────────────────●─────● ← always stable
│ ↑
feature ●───●───●───●───merge
↑ ↑

PULL REQUEST Create a Pull Request — Professional Team Workflow ›

A Pull Request (PR) is how teams review code before merging to main. Even solo projects benefit — it creates a record of every change and why it was made.

🖱️
1. Push your feature branch to GitHub: git push origin feature/add-docker
2. Go to your repo on GitHub — you'll see a yellow banner: "feature/add-docker had recent pushes" → click Compare & pull request
3. Add a title: "Add Docker support for TaskAPI"
4. In description write what you changed and why
5. Click Create pull request
6. Review the changes → click Merge pull request → Confirm merge
7. Delete the branch (GitHub shows a button)
💡
On your resume: "Used GitHub Pull Requests and code review workflow for all feature development". This one line shows you work like a professional team, not a solo hacker.
GITIGNORE .gitignore — Don't Push Secrets or Junk to GitHub ›

A .gitignore file tells Git: "never track these files". Critical for keeping secrets and build files off GitHub.

.gitignore for a Python DevOps project
# Python
__pycache__/
*.pyc
.venv/
venv/
# Secrets — NEVER commit these
.env ← API keys, passwords
*.pem ← SSH private keys
kubeconfig ← K8s cluster access
secrets.yaml
# Terraform
.terraform/
*.tfstate ← may contain secrets
*.tfstate.backup
# OS files
.DS_Store ← Mac junk file
Thumbs.db ← Windows junk file
🚨
Never commit .env files or private keys to GitHub. Even private repos. Bots scan GitHub 24/7 looking for exposed API keys. If you accidentally push a secret — rotate it immediately (change the password/key), don't just delete the commit.

Undo in Git — Never Panic Again
Git never truly deletes anything. Every mistake is fixable. Here's how.
fixing common mistakes
# ── Unstage a file (added too soon) ────────────────────
$ git restore --staged secret.env # removes from staging, keeps file
# ── Discard changes in a file (go back to last commit) ─
$ git restore app.py # WARNING: changes permanently lost
# ── Fix the last commit message ────────────────────────
$ git commit --amend -m "Better message" # only if not pushed yet
# ── Undo last commit (keep changes) ────────────────────
$ git reset HEAD~1 # uncommits but keeps file changes
# ── Go back to specific commit ─────────────────────────
$ git log --oneline
a3f2c91 Latest commit
b4f1e3d Good commit I want to go back to
$ git checkout b4f1e3d # time-travel to that commit
# ── Safe undo after pushing (creates new "undo" commit) ─
$ git revert a3f2c91 # safest way to undo on shared repos
[main h9k2p3] Revert "Latest commit"
⚠️
git reset --hard permanently deletes uncommitted changes. git push --force can overwrite teammates' work. Never use these on shared branches. Use git revert instead — it undoes safely by adding a new commit.

Common Git Errors — Exact Fix
ERR 01 CONFLICT — Merge conflict ›
$ git merge feature/login
CONFLICT (content): Merge conflict in app.py
# Open app.py — you'll see conflict markers:
<<<<<<< HEAD
def login(): return "main version"
=======
def login(): return "feature version"
>>>>>>> feature/login
# FIX: edit file, keep the right code, delete the markers
$ git add app.py
$ git commit -m "Resolve merge conflict in login"
ERR 02 Permission denied (publickey) ›
$ git push origin main
git@github.com: Permission denied (publickey)
# FIX: SSH key not added to GitHub
$ cat ~/.ssh/id_ed25519.pub # copy this output
# Paste at github.com → Settings → SSH keys → New SSH key
ERR 03 rejected — remote contains work you don't have ›
$ git push origin main
! [rejected] main -> main (fetch first)
# Someone else pushed to GitHub before you
# FIX: pull first, then push
$ git pull origin main # get their changes
$ git push origin main # now push yours

Git Interview Questions — Perfect Answers
Q1What is the difference between git merge and git rebase?Medium
git merge takes two branches and creates a new "merge commit" combining them. The history shows both branches. Safe for shared branches — never rewrites history.

git rebase moves your commits to the tip of another branch, rewriting history to look like they were always there. Result: linear, clean history. But it changes commit hashes — dangerous on shared branches.

Rule of thumb: merge for shared branches (main, develop), rebase for local feature branches before merging to keep history clean.
🎯 Say: "I use merge for main branch integration and rebase to clean up my local feature branch history before creating a PR. I never rebase shared branches."
Q2What is git stash and when would you use it?Easy
git stash temporarily saves your uncommitted changes to a stack and reverts your working directory to the last commit — without making a commit. Use it when: you're in the middle of work and need to urgently switch to another branch, or your manager says "hotfix this NOW" but you're halfway through a feature.

git stash — saves your work. git stash pop — restores it. git stash list — see all stashes.
🎯 Say: "git stash is my escape hatch when I need to switch context urgently. Stash, fix the hotfix, pop my stash, continue my feature."
Q3What is the difference between git pull and git fetch?Easy
git fetch downloads changes from GitHub to your local repo but does NOT merge them into your working branch. Safe to run anytime — your files don't change. Use it to preview what others pushed.

git pull = git fetch + git merge. Downloads AND immediately merges into your current branch. Can cause merge conflicts if you have local changes.

In CI/CD pipelines, fetch is preferred because you want explicit control over when merging happens.
🎯 Say: "I prefer git fetch + git diff to see what changed before merging. git pull is convenient for solo projects but git fetch gives you more control in team settings."
Q4What is GitOps?Hard
GitOps is a way of doing DevOps where Git is the single source of truth for everything — application code AND infrastructure configuration. Instead of manually running kubectl apply or terraform apply, you commit the desired state to Git. An automated tool (ArgoCD, Flux) watches the repo and automatically syncs the cluster to match what's in Git.

Benefits: every change is auditable (who changed what, when, why). Rollback = revert a commit. Disaster recovery = apply the repo. Consistency across environments. Everything is peer-reviewed via PRs before reaching production.
🎯 Say: "GitOps means git push IS the deployment. I practise this with GitHub Actions — a push to main automatically triggers test, build, and deploy. The cluster state is always what's in the repo."
Q5How do you handle a situation where you accidentally committed a secret/password?Medium
Two situations: Not pushed yet: git reset HEAD~1 to undo the commit, then add the file to .gitignore and recommit without it.

Already pushed to GitHub: The secret is compromised — assume it's been seen. Step 1: Rotate the secret immediately (change the password/API key — the old one is now invalid). Step 2: Remove from code, add to .gitignore. Step 3: Use git filter-branch or git-filter-repo to remove from history (though this is a complex operation on shared repos). Step 4: Force push. Step 5: Notify your team.
🎯 Say: "Prevention first — I use .gitignore and environment variables. If it happens, rotate first, ask questions later. I use GitHub Secrets for CI/CD credentials — they never touch the codebase."
