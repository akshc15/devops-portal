// why monitoring exists
The Story — Finding Out From Twitter
🐦 The worst way to find out your app is down — a tweet from an angry user.

Before monitoring, teams found out about outages when customers complained. By the time they investigated, dozens of errors had happened, data was corrupted, and the business had lost money. Debugging a dead system is like reading an obituary — the patient is already gone.

Google's SRE team coined the concept of "observability" — the ability to understand the internal state of a system from its external outputs. They defined the Four Golden Signals — latency, traffic, errors, saturation — as the minimum metrics every production system must track.

Today, Prometheus + Grafana is the de-facto open-source monitoring stack used by thousands of companies. It runs on Kubernetes natively, costs nothing, and can handle millions of metrics per second.
🧒 Explain Like I'm 5

Imagine you have a car. You drive without looking at the dashboard — no speedometer, no fuel gauge, no temperature warning. Everything feels fine until you're stranded on a highway with an overheated engine and empty tank. Monitoring is your car dashboard — it tells you what's happening inside while everything still looks fine from the outside. Prometheus is the sensors. Grafana is the dashboard screen.

The Four Golden Signals — Google's Framework
Measure these 4 things for any system and you'll know if it's healthy.
⏱️
Latency
How long does a request take? Track p50, p95, p99 — not just average. A spike means users are waiting.
http_request_duration_seconds
🚦
Traffic
How many requests per second? Know your baseline. Sudden drops = something broke. Spikes = scaling needed.
http_requests_total
🔴
Errors
What fraction of requests fail? Alert when error rate exceeds 1%. Even 0.1% errors = 1 in 1000 users failing.
http_errors_total
📈
Saturation
How full is the system? CPU%, memory%, queue depth. 80% CPU is fine. 95% CPU means trouble soon.
container_cpu_usage
The 3 Pillars of Observability
📈
Metrics
Prometheus
"What is happening?"
Numbers over time. CPU%, request rate, error count.
📜
Logs
Loki / ELK
"Why did it happen?"
Detailed records of every event. Error messages, stack traces.
🔍
Traces
Jaeger / Tempo
"Where did it happen?"
Follow a request through every service. Find the slow step.
💡
Start with metrics. Logs tell you what happened AFTER the fact. Metrics tell you what's happening RIGHT NOW and trending. You can alert on metrics. You can't alert on logs efficiently. Master Prometheus + Grafana first, then add Loki for logs.

// how prometheus works
Prometheus — Collecting Metrics
How Prometheus works:

Prometheus is a pull-based system — it doesn't wait for your app to send metrics. Instead, it visits your app's /metrics endpoint every 15 seconds and scrapes the numbers. Your app just needs to expose metrics at that endpoint. Prometheus stores them as time-series data (value + timestamp).

The metrics format looks like: http_requests_total{method="GET",status="200"} 1234. A metric name, optional labels in {}, and a value. Prometheus stores this every 15 seconds and you can query it with PromQL.
promql — prometheus query language
# ── Basic queries ──────────────────────────────────────
# Total HTTP requests
http_requests_total
# Requests for a specific endpoint
http_requests_total{path="/health", method="GET"}
# Request RATE per second (over last 5 minutes)
rate(http_requests_total[5m])
# Error rate (5xx responses)
rate(http_requests_total{status=~"5.."}[5m])
# CPU usage for all taskapi pods
container_cpu_usage_seconds_total{pod=~"taskapi-.*"}
# 95th percentile latency (p95)
histogram_quantile(0.95, rate(http_request_duration_seconds_bucket[5m]))
# Memory usage in MB
container_memory_usage_bytes{pod=~"taskapi-.*"} / 1024 / 1024

Add /metrics to TaskAPI — 3 Lines of Code
add prometheus metrics to fastapi
# Install the library
$ pip install prometheus-fastapi-instrumentator
# Add 3 lines to app/main.py:
from fastapi import FastAPI
from prometheus_fastapi_instrumentator import Instrumentator ← add this
app = FastAPI()
Instrumentator().instrument(app).expose(app) ← add this

tell prometheus to scrape your app — servicemonitor.yaml
apiVersion: monitoring.coreos.com/v1
kind: ServiceMonitor
metadata:
name: taskapi-monitor
namespace: monitoring
labels:
release: monitoring # must match Prometheus selector
spec:
selector:
matchLabels:
app: taskapi
endpoints:
- port: http
path: /metrics
interval: 15s # scrape every 15 seconds

Alerting — PrometheusRule + AlertManager
prometheusrule.yaml — alert when error rate is high
apiVersion: monitoring.coreos.com/v1
kind: PrometheusRule
metadata:
name: taskapi-alerts
namespace: monitoring
labels:
release: monitoring
spec:
groups:
- name: taskapi
rules:
# Alert if error rate > 5% for 5 minutes
- alert: HighErrorRate
expr: |
rate(http_requests_total{status_code=~"5.."}[5m])
/
rate(http_requests_total[5m]) > 0.05
for: 5m # must be true for 5 mins to fire
labels:
severity: critical
annotations:
summary: "TaskAPI error rate above 5%"
# Alert if pod is not running
- alert: PodDown
expr: kube_pod_status_ready{pod=~"taskapi-.*"} == 0
for: 1m
labels:
severity: warning
# Alert if disk > 80%
- alert: DiskSpaceHigh
expr: (node_filesystem_size_bytes - node_filesystem_free_bytes) / node_filesystem_size_bytes > 0.8
for: 10m
💡
Send alerts to Slack (free): AlertManager can send alerts anywhere. Create a Slack webhook at api.slack.com/apps → Incoming Webhooks. Add the webhook URL to AlertManager config. Now production alerts land in your Slack channel with context and severity.

Monitoring Interview Questions
Q1What are the Four Golden Signals?Easy
From Google's SRE book — the minimum set of metrics for any service:

1. Latency: Time to serve a request. Track p50/p95/p99 (median and tail latency), not just average. A slow request is often worse than a failed one (user waits and gets nothing).
2. Traffic: Demand on your system — requests per second, transactions per minute. Establishes baseline for capacity planning.
3. Errors: Rate of failed requests — explicit failures (500s) and implicit (200 with wrong content). Target: less than 0.1% error rate.
4. Saturation: How "full" your service is — CPU%, memory%, I/O wait. Most services degrade before hitting 100%.
🎯 Say: "My dashboards always start with the four golden signals. For TaskAPI: Grafana shows request rate, p99 latency, error rate, and CPU saturation on the first screen. That tells me if something is wrong in 5 seconds."
Q2What is the difference between monitoring and observability?Medium
Monitoring = tracking known failure modes. You define what to measure upfront (CPU, error rate). You get alerts when known thresholds are breached. Works well for systems you understand completely.

Observability = the ability to understand ANY internal state from external outputs — including unknown failure modes. Achieved through three pillars: metrics (what), logs (why), traces (where). An observable system lets you debug novel failures without needing to add new instrumentation.

Analogy: monitoring is checking your car's known warning lights. Observability is having enough sensors to diagnose a problem that's never occurred before.
🎯 Say: "Monitoring tells you that something is wrong. Observability tells you why. I start with Prometheus metrics for monitoring, add structured logs with Loki, and use distributed tracing for complex microservice debugging."
Q3What is an SLI, SLO, and SLA?Hard
SLI (Service Level Indicator): A metric that measures service quality. Examples: request success rate, latency p99, availability percentage. Raw numbers.

SLO (Service Level Objective): A target for an SLI. "99.9% of requests succeed" or "p99 latency <200ms". Internal goal — not contractual. Your team is accountable to SLOs.

SLA (Service Level Agreement): A legal contract with customers about service quality. If SLA is violated, there are financial consequences (credits, refunds). SLA is always less aggressive than SLO — you need buffer between your internal objective and external commitment.

Relationship: SLI is measured → compared to SLO → SLA is the external commitment based on SLO history.
🎯 Say: "Our SLO for TaskAPI is 99.5% availability with p99 latency under 300ms. We alert when we're burning error budget too fast — before we breach the SLO. The SLA with customers is 99% — giving us buffer."
